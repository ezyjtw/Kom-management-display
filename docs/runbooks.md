# Operational Runbooks

## 1. Database Outage Recovery

### Symptoms
- `/api/health/readiness` returns 503
- `/api/health` shows `database.connected: false`
- Application logs: `PrismaClientKnownRequestError: Can't reach database server`

### Triage Steps

1. **Verify the database is actually down**
   ```bash
   # Check PostgreSQL process
   docker compose ps db
   # Or check Railway DB status in dashboard

   # Test connectivity directly
   psql $DATABASE_URL -c "SELECT 1"
   ```

2. **Check disk space**
   ```bash
   docker exec <db-container> df -h /var/lib/postgresql/data
   ```

3. **Check PostgreSQL logs**
   ```bash
   docker compose logs db --tail=100
   ```

### Resolution

**If database is stopped:**
```bash
docker compose up -d db
# Wait for health check to pass
docker compose ps db  # should show "healthy"
```

**If database is out of disk:**
1. Identify and remove large unused tables or run VACUUM FULL
2. Increase volume size if cloud-hosted
3. Review retention policy for data that should have been purged

**If database is corrupted:**
1. Stop the application: `docker compose stop app`
2. Restore from latest backup:
   ```bash
   pg_restore -h <host> -U <user> -d kommand --clean < backup.sql
   ```
3. Restart the application: `docker compose up -d app`
4. Verify data integrity via `/api/health`
5. Check for data loss window (RPO is 24 hours)

### Post-incident
- Record the outage in the incident tracker
- Verify audit log continuity (no gaps)
- Run integration health check to ensure all connectors reconnected

## 2. Bad Deployment Rollback

### Symptoms
- New deployment shows increased error rate (5xx responses)
- Health checks failing after deploy
- Users reporting broken functionality

### Immediate Actions

1. **Confirm the issue is deployment-related**
   ```bash
   # Check health
   curl -s https://<domain>/api/health | jq .

   # Check recent logs for errors
   docker compose logs app --tail=200 | grep ERROR
   ```

2. **Rollback the application**

   **Railway:**
   ```bash
   railway rollback
   ```

   **Docker:**
   ```bash
   # Find previous working image
   docker images kommand-centre --format "{{.Tag}} {{.CreatedAt}}"

   # Deploy previous version
   docker stop kommand-centre
   docker run -d --name kommand-centre -p 3000:3000 \
     --env-file .env kommand-centre:<previous-tag>
   ```

3. **If a database migration was applied:**
   - Do NOT rollback the migration unless the old code cannot work with the new schema
   - If rollback is necessary, restore from pre-deployment backup
   - Coordinate migration rollback and code rollback together

### Post-rollback
- Verify health checks pass
- Verify login and critical flows work
- Investigate root cause before re-deploying
- Add regression test for the failure

## 3. Integration Connector Failure

### Symptoms
- `/api/integrations/health` shows one or more adapters as `down` or `degraded`
- Thread ingestion has stopped (no new threads from that source)
- Queue backlog growing

### Triage Steps

1. **Check integration health**
   ```bash
   curl -s -H "Cookie: <admin-session>" https://<domain>/api/integrations/health | jq .
   ```

2. **Identify which adapter is failing** -- look for `status: "down"` or high `failureCount`

3. **Common causes and fixes:**

   | Cause | Fix |
   |-------|-----|
   | API credentials expired | Rotate credentials, update env vars, restart |
   | External service outage | Check provider status page, wait for recovery |
   | Rate limit exceeded | Wait for rate limit window to reset (check `rateLimitRemaining`) |
   | Network/DNS issue | Check container DNS resolution, restart if needed |
   | Schema change in external API | Review adapter code against new API docs |

4. **Manual sync retry** (after fixing the cause):
   ```bash
   # Trigger manual sync for the affected adapter
   curl -X POST -H "Cookie: <admin-session>" https://<domain>/api/integrations/<adapter>
   ```

5. **Check dead-letter queue** for events that failed max retries:
   - Review failed events in the job queue
   - Fix the root cause
   - Replay dead-letter events

### Post-incident
- Verify backlog is draining
- Check for duplicate events after replay
- Update monitoring thresholds if alerting was too slow

## 4. SLA Breach Triage

### Symptoms
- Alerts generated by `POST /api/alerts/generate` (cron job)
- Dashboard showing SLA breach warnings
- Thread age exceeding configured thresholds

### Triage Steps

1. **Review active alerts**
   ```bash
   curl -s -H "Cookie: <admin-session>" https://<domain>/api/comms/alerts | jq '.data'
   ```

2. **Identify breached threads** -- check thread age, owner, and status

3. **Determine cause:**

   | Scenario | Action |
   |----------|--------|
   | Thread unassigned | Assign to available team member |
   | Owner on leave | Transfer ownership to backup |
   | Waiting on external party | Add note, extend SLA if policy allows |
   | Thread misclassified | Update priority/category, reassign |
   | Legitimate breach | Escalate to team lead, document reason |

4. **Resolve the alert**
   - Address the underlying thread (respond, reassign, or resolve)
   - Mark the alert as resolved via the UI or API

### Prevention
- Review SLA thresholds quarterly
- Ensure coverage during team member absences (ownership transfer)
- Monitor alert generation cron is running on schedule

## 5. Incident Response Workflow

### Severity Levels

| Severity | Criteria | Response Time | Examples |
|----------|----------|---------------|----------|
| P1 - Critical | Platform down, data loss risk | 15 minutes | DB outage, auth system failure |
| P2 - High | Major feature broken, data integrity concern | 1 hour | Integration failure, scoring errors |
| P3 - Medium | Degraded functionality, workaround available | 4 hours | Slow queries, UI bugs |
| P4 - Low | Minor issue, cosmetic | Next business day | Typos, non-critical UI issues |

### Response Steps

1. **Detect**: Via health checks, monitoring alerts, or user reports
2. **Acknowledge**: Assign an incident owner within response time SLA
3. **Diagnose**: Use health endpoints, logs, and integration status to identify root cause
4. **Mitigate**: Apply immediate fix (rollback, restart, config change) to restore service
5. **Resolve**: Deploy permanent fix after root cause analysis
6. **Document**: Record in incident tracker:
   - Timeline of events
   - Root cause
   - Impact (users affected, data affected, duration)
   - Corrective actions
   - Preventive measures

### Communication
- P1/P2: Notify all stakeholders immediately via Slack
- P3/P4: Update in next standup
- All: Post-incident review within 48 hours for P1/P2

## 6. Export Audit Investigation

### When to Investigate
- Suspicious export activity flagged in audit logs
- Large data exports by unusual users
- Export of sensitive resources (travel rule cases, audit logs)

### Investigation Steps

1. **Query the audit log for export events**
   ```bash
   curl -s -H "Cookie: <admin-session>" \
     "https://<domain>/api/audit?action=export&dateFrom=2024-01-01&dateTo=2024-01-31" | jq .
   ```

2. **Review each export record:**
   - `userId`: Who performed the export
   - `resource`: What data was exported
   - `filters`: What scope/filters were applied
   - `rowCount`: How many records were exported
   - `timestamp`: When the export occurred
   - `requestId`: Correlation ID for full request trace

3. **Cross-reference with user permissions:**
   - Verify the user's role permits export for that resource
   - Check if the scope filter was correctly applied (team vs all)
   - Verify the user was active and authorized at the time

4. **Check for anomalies:**
   - Exports at unusual hours
   - Exports significantly larger than typical
   - Multiple exports in quick succession
   - Exports immediately before/after role changes

### Escalation
- If unauthorized access is confirmed, disable the user account immediately
- Preserve all audit logs related to the incident
- Report to compliance team per data breach procedures
